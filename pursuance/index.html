Task Assignment

Mental model: Tasks as cards, sent to participants. They then have a deck to work with.

Fields

  • Title input, required
  • Description textarea, possibly markdown, checklist completion as progress metrics
  • Manager user list, defaults to host. defines who assigns and reviews Task
  • Volunteer user list, default blank. defines who accomplishes it
  • Deliverable? default is NO (Deliverables are outside the System, and just Manager Approval is enough), if yes, File Upload is unlocked (define specs: picture, pdf, word document, text document)
  • Sequential? default is NO. if YES, opens up new form asking Parent Task (Task can only start after Parent Task is Approved) or Child Task (Task approval unlocks Child Task)
  • Forum like Main Forum, and open to all Volunteers. think channel

Phases

  1. Unassigned Task is not paired with a Volunteer
  2. In Progress Task is paired with a Volunteer. depending on project pacing, it nags Volunteer to finish it, via Notification
  3. On Hold Task is paired with a Volunteer, but has a Parent Task that is not Approved
  4. Done Volunteer follows requirements, deem Task completed, and hits Done manually. Manager receives a Notification to review it. If Manager disagrees it's done, Task returns to In Progress, with a note (in forum). If Manager agrees it's Done, it's Approved
  5. Approved Task is approved, System congratulates Volunteer and Manager on Main Forum (plus Notification). If Task has Children, Child Tasks move automatically from On Hold to In Progress, and both Volunteers and Managers are notified.
